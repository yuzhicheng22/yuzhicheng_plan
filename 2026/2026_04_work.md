### + 阅读文章《2016 年，我做过一次 AI 写代码创业》 #read #blog

文章讲述了作者在2016年从 google 辞职创业，全身心投入 ai code 方向，却因当时谁都不愿意为一个还没有被时代证明的愿景下注而遇到阻碍，哪怕方向是正确的，却也因现实因素而终止了。
作者在2026年的当下对这个创业项目进行复盘，看看他做对了什么，又做错了什么：

做错了什么：
- 在寻找投资这个事情上走错了方向，试图说服所有人，而不是尽快找到少数本来就能懂的人；
- 作者在这件事情上想完成一个完善的版本后再发布，而不是 build in public，这与创业环境有关，在开发的过程中积累势能；
- 没清晰的知道自己的 failure mode;

做对了什么：
- 没有让想法仅仅停留在想法，而是真的去做了，并且转化自己人生经历的一部分；
- 坚定自己的立场，明确自己的目标，没有被投资者裹挟走上和自己设想道路不同的方向，即使付出了一些代价，但也争取到了一定的自由；

同时作者还在文章中提到 Illia Polosukhin 的故事，他实际上是 Transformer 论文《Attention Is All You Need》的作者之一，现如今的 ai 时代几乎就建立在 Transformer 这项工作之上。

当时这位老哥正在和朋友一起做一项关于 ai 生成 ACM 竞赛回答代码的项目，在拿到投资知道创立了 nearai 这家公司，拿到投资后和作者设想的线性技术路线不同，这家公司最终彻底变成了一家区块链公司。

这个故事的感慨之处在于：

> 我们这些身处其中的人，其实一直都被未来的迷雾包围着。所谓“看对了方向”或者“看错了方向”，很多时候并不足以决定你最后会走到哪里。你也许看见了未来，却没有资源；你也许拥有资源，却走向了别的方向；你也许参与了最关键的基础工作，却仍然不知道它会在几年后如何改变世界。

未来并不是线性展开的。这对当下 ai 浪潮过于声势浩大的时代下的普通人和互联网从业者，也许可以缓解一些焦虑。

### * 开发个人生活记录app，基于 John Carmack 的 `.plan` 理念 #idea #blog #project/plan

我很喜欢现在这种记录的方式，我想也许可以实现一款个人工作/想法/日常的记录性质的软件；
目前我的主要以 markdown 格式进行记录，更现代化，更容易阅读，使用不同标题层级进行区分，由于我的记录密度并不高，所以采用以月单位进行记录，这样维护一个很长的文件也让我很享受。
并且我的文件主要托管在github上，对于普通人或者非程序员来说，使用github并不那么便利。
由于依旧保留了纯文本，并且语法尽可能的简单，基本只包含了 markdown 的标题、无序/有序列表、波浪线，所以编辑文件我可以在网页，或者 Obsidian 上进行，理论上任何编辑器甚至微信输入框都可以成为输入；

核心理念是做一个让普通人也能轻松维护“自己的 Carmack .plan” 的 App —— 极简、纯文本、列表式、持续单一文件感、真实、多端同步。

- **极致低摩擦 + “不追求完美”**：记录应该像打开一个文本文件随便写几行一样简单，允许“脏”的记录、吐槽、半成品想法、甚至错别字。核心是真实倾倒脑内内容，而不是创作精美笔记。
- **纯文本优先 + 列表式记录**：核心格式是纯文本（支持 Markdown 渲染），常用符号如 +（已完成）、-（待办）、*（想法/备注）。这让记录既有结构，又保持原始感。
- **持续的单一“文件”感觉**：流动的记录，想法，日志抛开所有压力，只需要流动在文件中；
- **简单即强大（“worse is better” 精神）**：避免功能膨胀，只做好记录核心，后续加财务、公开、社交时，不能让主流程变复杂。

目标人群：就能吸引对 Carmack 风格有共鸣的用户（开发者、知识工作者、想养成复盘习惯的人）。
关键词：极简、纯文本、列表式、真实不完美、持续单一文件感、低摩擦记录。

### + 竞品的对比 flomo 和 mynd #project/plan

由于`.plan`这个软件的名字我还没有想好，我们就以 plan 为代号；

mynd: https://myndnote.app/

mynd 这个软件的作者是一名 vloger， 在毫无变成和项目开发的背景下，配合 vibe coding 完成了 mynd 这个软件，的确很有想法，采用了聊天消息短信的形式进行记录，同时软件的使用也非常的简单，轻量。启动就是聊天输入框，像给自己发微信一样记录。没有多余页面、没有复杂导航。这和我“低摩擦、不追求完美、轻松记录”的核心高度一致。
cbvivi 用 vibe coding 完成整个 App，证明了高度 AI 协同开发完全可行。
有几个点我值得关注：

1. 目前仅支持 ios 平台，因为据开发者所说计划是先全力完善 iPhone 版，所以开发过程中先实现比考虑太多更重要，特别是使用 vibe coding 产出的作品；
2. 关于 AI，mynd 的态度是保持距离，面向未来。同时 markdown 文档也天然适合想要利用大模型来分析和总结的用户。
3. 文件高于应用的理念：mynd 的消息会存储在一个日期命名的 markdown 文件中，通过标签和 markdown 语法来进行区分，同时每一条消息使用注释一个 uid 进行标记，而应用本身更像是一个根据作者意图和设计来展现文件的“壳”；

```
# 今天一般般 - 17:45 #trash(2026-04-14) <!-- id:a48f9514 -->

测试1 - 19:52 <!-- id:b00380b0 -->

- [ ] 测试2 - 19:52 <!-- id:d723cd88 -->

**测试3** - 19:52 #right <!-- id:65d7e769 -->

测试4 - 19:52 <!-- id:f2ecb749 -->

测试5 - 19:58 🎈 <!-- id:b1b4b6ea -->

> 测试6 - 19:59 <!-- id:d789ccb5 -->

![photo](media/1DB7D8D6-07C2-409A-B4F9-A3CC8A6C9CE9.jpg) - 19:59 <!-- id:ae670557 -->
```

flomo（浮墨笔记） 是一款以卡片式极简记录为核心的笔记 App，和前面的 mynd 不同，这是一家成熟的商业公司（上海先帝网络科技有限公司）的产品，用户基数大，当然，产品的设计，以及完成度也要高很多；
flomo 深受《卡片笔记写作法》（Luhmann 卡片盒理念）影响，定位为“流动的 memo”（flomo = flow + memo），强调无压力捕捉碎片想法，而非复杂文章写作，这点和我设想的软件在呈现上几乎是两个方向。并且 flomo 也几乎能做到我设想中 app 能做到的一切事情，标签、时间轴记录、回顾、待办，区别在于 flomo 使用卡片式 Memo + 多级标签（#标签 / 子标签）进行呈现，十分方便检索和回顾；

首先不可否认的是 flomo 是一个很成熟，功能很强大的软件，但也有一些差异，很多差异是有意为之（出于成熟商业公司的考量）：

- 数据存储：卡片式后端存储（类似结构化 Memo），而非纯物理单个 Markdown 文件，同步依赖自有云 + 多平台支持，但支持导出为 html 格式。
  对比这点 plan 的数据存储格式还没有构思好，但存储上是依赖本地存储的纯文本，同步依赖 icloud（具体为 SwiftData + CloudKit），数据对于用户来说更可控，对开发者来说成本更可控。
- 呈现形式和用户习惯：这点目前来说 plan 的呈现和记录体验都是基于 John Carmack 的 `.plan` 理念进行开发，不能说哪种就一定更好更科学，但 plan 的呈现方式也许更适合文字工作者、开发者进行严肃的工作记录，plan 强调“不追求完美、真实倾倒” —— flomo 更偏知识积累，plan 偏个人真实复盘 + 生活工作混合。
- 多平台 + 小组件：flomo 全平台经验值得学习

### * 思考关于 plan 软件的必要性 #blog #progject/plan

我想：如果不需要这个按月/周/天呈现的功能，几乎只需要使用纯文本md来存储，并且创建文件以及存储周期都由用户决定，这样似乎更符合文件大于应用的理念，但这样一来的话，似乎任何一个支持纯文本编辑的软件配合文件夹应用都能实现，是否失去了意义，我能做的只是推广这个plan记录的方法，以及优化记录体验，因为我目前就是是用obsidian来进行记录的。

我是否在将一个简单的记录过程复杂化；

目前已经有很多用户根据自身的需求在 obsidian 中实现了更适合自己的工作流，很难做出让 Obsidian 用户迁移的理由，如果我非要用一个软件去专注在这种记录模式上我能做的差异只有：

- 专为列表式 .plan 优化的编辑体验（Obsidian 默认没有）：
  - 快捷插入 + 、- 、* （工具栏或手势）。
  - 实时高亮列表符号、软换行优化、专注模式（隐藏侧边栏，只剩纯文本 + 禅意背景/字体）。
- Apple 生态原生舒适感（Obsidian 在 iOS 上仍较笨重）：
  - 完美的 iOS + macOS 统一体验（Handoff、Continuity、动态岛、小组件随机显示历史片段）。
  - 更流畅的 Markdown 实时渲染 + 预览分栏（可选）。
  - 更小的 App 体积、更快的启动、更省电（尤其适合每天多次快速记录）。
- App 内内置 Carmack .plan 理念介绍

即使去掉自动聚合呈现，依然可以围绕 Carmack .plan + 流动禅 + 心流 + 低摩擦 打造专属体验，让它不是“又一个 Markdown 编辑器”，而是“专为 .plan 式记录而生的极简伴侣”。关键在于优化记录的过程和心理感受，而非底层文件。

思考过多意义也许只会让我畏手畏脚，毕竟有这个想法的起始点除了想分享这种自己享受其中的记录方式之外，还是想尝试一下 vibe coding，试试先去做吧。
